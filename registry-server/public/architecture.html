Architecture · trust boundaries

Four routes.
No hidden relay.

TETHER separates browser conversation traffic, public selector updates, privacy-safe drift reports, and privileged maintenance work. Each route gets only the access it needs.

A request never needs the registry.

The coding harness talks to a loopback service on your machine. The extension owns one selected browser tab and completes the request through that existing provider session.

Route 02 · public reads

Selector updates are data, never code.

The extension periodically checks a public registry index. It downloads only changed provider manifests, verifies their origin and integrity, tests them without taking action, and keeps a safe fallback.

About the SHA-256 values

They are public checksums—not passwords, API keys, or signing secrets. The extension needs them to prove that a downloaded manifest matches the version published by TETHER.

Route 03 · bounded writes

Failures become maintenance signals.

When a packaged selector repeatedly stops matching, the extension can send a small allowlisted report. It never uploads prompts, responses, page HTML, screenshots, cookies, or account identity.

  • AcceptedOrigin, adapter version, extension version, error category
  • RejectedUnknown fields, unknown origins, oversized bodies
  • ProtectionRate limits, deduplication, strict schema validation

Route 04 · authentication

JWT belongs on privileged routes—not inside a public extension.

A shared JWT secret packaged into Chrome would be extractable by every installer. TETHER therefore uses public verification for public data and reserves short-lived tokens for server-to-server or per-session local authority.

Security rule

Long-lived database credentials, Redis passwords, signing private keys, and JWT signing keys exist only in backend environment variables or CI secrets. They are never returned by these public pages or registry routes.

Failure behavior

Updates can fail.
Your extension should not.

The extension accepts a remote manifest only after bounded validation. Otherwise it keeps the previous working selectors or returns to its packaged provider definition.

01

No remote code

Downloaded files contain selectors and structural fingerprints only. Automation behavior stays inside the reviewed extension bundle.

02

Automatic fallback

If an update is invalid or stops matching the page, TETHER keeps the previous working selector set.

03

Auditable rollback

Every provider change is tied to Git history, reviewable diffs, validation, and a reversible registry version.
